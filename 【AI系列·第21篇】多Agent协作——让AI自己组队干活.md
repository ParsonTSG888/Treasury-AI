【AI系列·第21篇】多Agent协作——让AI自己组队干活

文 / Parson资金研习社

---

前面 20 篇，一直在用"一个 AI 干一个活"的模式——你给指令，AI 执行，你检查结果，再给下一个指令。

这个模式在简单任务上没问题。但遇到复杂任务——比如"同时审核 10 份合同、生成汇总报告、更新知识库、通知相关人员"——一个 AI 串行执行，效率太低。

**能不能让多个 AI 同时干活？**

能。这就是 Claude Code 的多 Agent 协作能力。

---

## 一、从"一个员工"到"一个团队"

先打个比方。

你是资金部经理，手里有一个实习生（单 Agent 模式）。你让他做资金日报，他一步步来：查数据 → 清洗 → 汇总 → 出报告。每一步你都要确认，整个流程 2 小时。

现在你有 5 个实习生（多 Agent 模式）。你把任务拆开：小王查子公司 A-C 的数据，小李查 D-F 的数据，小张做数据清洗，小赵出报告，小刘发通知。5 个人同时干，20 分钟搞定。

**多 Agent 协作的本质：把一个大任务拆成多个小任务，分配给多个 AI 并行执行。**

在 Claude Code 里，这叫 **Agent Teams（团队模式）** 和 **Sub-agents（子 Agent）**。

![多 Agent 协作：从"一个员工"到"一个团队"](../参考图片/multi-agent.png)

## 二、Claude Code 的两种多 Agent 模式

### 模式一：子 Agent（Sub-agent）

主 Agent（你正在对话的 Claude Code）可以"派出"子 Agent 去执行特定任务。

```
你：帮我同时检查这 5 份合同的风险条款。

主 Agent：我将派出 5 个子 Agent 并行检查。

  子 Agent 1：检查合同 A（银行授信合同）
  子 Agent 2：检查合同 B（供应商协议）
  子 Agent 3：检查合同 C（经销商协议）
  子 Agent 4：检查合同 D（信用证）
  子 Agent 5：检查合同 E（保函）

  [5 个子 Agent 同时运行，各自返回结果]

主 Agent：汇总 5 份合同的风险检查结果，生成汇总报告。
```

**子 Agent 的特点：**
- 每个子 Agent 有独立的上下文窗口（互不干扰）
- 子 Agent 之间不直接通信，都通过主 Agent 协调
- 子 Agent 完成任务后自动销毁，不占用资源

**适用场景：** 批量处理、并行检查、多文件操作。

### 模式二：Agent Teams（团队模式）

更高级的协作模式。多个 Agent 组成一个团队，有明确的分工和协调机制。

在 Claude Code 的配置里启用：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

团队模式下，Agent 之间可以：
- 共享工作区（通过 Git Worktree 隔离）
- 互相传递中间结果
- 由"协调者"Agent 统一调度

**适用场景：** 复杂的多步骤任务，需要不同 Agent 处理不同环节。

## 三、Git Worktree：给每个 Agent 一个独立空间

多 Agent 并行工作的最大问题是**冲突**——两个 Agent 同时修改同一个文件怎么办？

Claude Code 用 **Git Worktree** 解决这个问题。

Git Worktree 是 Git 的原生功能——一个仓库可以有多个"工作目录"，每个目录是独立的，互不干扰。

```
主仓库/
├── worktree-agent-1/    ← 子 Agent 1 的工作目录
│   ├── contract-a.md
│   └── analysis-a.md
├── worktree-agent-2/    ← 子 Agent 2 的工作目录
│   ├── contract-b.md
│   └── analysis-b.md
└── worktree-agent-3/    ← 子 Agent 3 的工作目录
    ├── contract-c.md
    └── analysis-c.md
```

每个子 Agent 在自己的 Worktree 里工作，不会影响其他 Agent。所有任务完成后，主 Agent 把各个 Worktree 的结果合并回主仓库。

**类比：** 每个实习生有自己的工位和文件夹，不会互相拿错文件。工作完成后，把所有文件交给你统一归档。

## 四、权限模式：Plan / Auto / Bypass

多 Agent 协作时，权限管理变得更重要。Claude Code 有三种权限模式：

### Plan 模式（规划模式）

AI 先列出执行计划，你确认后才执行。

```
你：帮我生成这个月的资金分析报告。

AI（Plan 模式）：
执行计划：
1. 查询所有子公司本月的资金流水（预计 30 秒）
2. 按子公司分组汇总（预计 10 秒）
3. 计算关键指标（流动比率、资金集中度）（预计 15 秒）
4. 生成分析报告（预计 20 秒）
5. 上传到 WPS 知识库（预计 5 秒）

是否确认执行？[Y/n]
```

**适合：** 高风险操作、你不确定 AI 会怎么做的场景、首次使用新功能。

### Auto 模式（自动模式）

AI 直接执行，不需要你逐步确认。

```
你：帮我生成这个月的资金分析报告。

AI（Auto 模式）：
[直接开始执行，完成后展示结果]
```

**适合：** 你信任 AI 的操作、低风险任务、批量处理。

### Bypass 模式（绕过模式）

跳过所有安全检查，直接执行。包括文件写入、系统命令等。

**适合：** 极少数场景，比如自动化脚本的无人值守运行。

**建议：日常使用 Plan 模式，熟悉后切 Auto 模式。Bypass 模式慎用。**

三种模式的对比：

| 维度 | Plan | Auto | Bypass |
|------|------|------|--------|
| 确认频率 | 每步确认 | 首次确认 | 不确认 |
| 安全性 | 最高 | 中等 | 最低 |
| 效率 | 最低 | 中等 | 最高 |
| 适合场景 | 高风险/首次 | 日常 | 无人值守 |

## 五、Effort Level：控制 AI 的"用力程度"

Claude Code 有一个隐藏的高级配置：**Effort Level**。

它控制 AI 在每个任务上投入多少"思考精力"。

```json
{
  "env": {
    "CLAUDE_CODE_EFFORT_LEVEL": "max"
  }
}
```

三个级别：

| 级别 | 说明 | 适合场景 | Token 消耗 |
|------|------|---------|-----------|
| `low` | 快速响应，浅层思考 | 简单查询、格式化 | 低 |
| `medium` | 中等深度 | 常规任务 | 中 |
| `max` | 深度思考，多步推理 | 复杂分析、架构设计 | 高 |

**怎么选？**

- 日常查数据、改格式 → `low`
- 生成报告、审核合同 → `medium`
- 跨系统架构设计、复杂财务模型 → `max`

**Effort Level 不是越高越好。** `max` 模式下，AI 会花更多时间"思考"，响应变慢，Token 消耗也更多。简单任务用 `max` 是浪费。

**建议：** 在 `~/.claude/settings.json` 里设置默认级别，需要时在对话中临时调整。

## 六、实战：用多 Agent 审核 10 份合同

把前面的概念串起来，看一个完整的实战场景。

### 场景

你是集团资金部的人，刚收到银行发来的 10 份授信合同（10 家子公司各一份）。需要在今天下班前完成审核，标出风险条款。

### 单 Agent 模式（传统方式）

```
你：帮我审核这 10 份合同。
AI：好的，我一份一份来。
  [审核合同 1...]
  [审核合同 2...]
  [审核合同 3...]
  ...
  [审核合同 10...]
总耗时：约 40 分钟。
```

### 多 Agent 模式（进阶方式）

```
你：帮我同时审核这 10 份合同，用子 Agent 并行执行。

主 Agent：
  派出 10 个子 Agent，每个审核一份合同。
  [子 Agent 1-10 同时运行]

  子 Agent 1：审核子公司 A 合同 → 发现 3 个风险条款
  子 Agent 2：审核子公司 B 合同 → 发现 1 个风险条款
  子 Agent 3：审核子公司 C 合同 → 无重大风险
  ...

  [所有子 Agent 完成，约 5 分钟]

主 Agent：
  汇总 10 份合同的审核结果，生成风险矩阵：

  | 子公司 | 风险条款数 | 最高风险等级 | 关键风险 |
  |--------|-----------|------------|---------|
  | A | 3 | 高 | MAC条款、交叉违约 |
  | B | 1 | 中 | 限制性财务指标 |
  | C | 0 | 低 | - |
  | ... | ... | ... | ... |

  详细审核报告已上传到 WPS 知识库。
```

**耗时：约 8 分钟（含汇总）。效率提升 5 倍。**

## 七、多 Agent 的注意事项

### 注意一：不是所有任务都适合并行

**适合并行的任务：**
- 批量处理（审核多份合同、处理多个 Excel）
- 多维度分析（同时查数据、同时生成图表）
- 独立子任务（每个子公司独立分析）

**不适合并行的任务：**
- 有前后依赖（先查数据，才能生成报告）
- 需要全局上下文（分析整体趋势需要看所有数据）
- 修改同一资源（多个 Agent 同时改同一个文件）

### 注意二：成本会增加

每个子 Agent 都消耗 Token。10 个子 Agent 并行，Token 消耗大约是单 Agent 的 8-10 倍（不是 10 倍，因为有些公共上下文可以复用）。

**建议：** 简单任务用单 Agent，复杂任务才上多 Agent。不要为了"看起来高级"而滥用。

### 注意三：结果需要人工复核

多 Agent 并行执行时，你很难实时监控每个 Agent 的中间过程。最终结果需要人工抽查——特别是涉及金额计算、风险判断的场景。

**建议：** 前几次使用时，把子 Agent 的输出全部保留，人工逐个检查。确认准确率后，再减少抽查频率。

## 八、配置速查表

```json
// ~/.claude/settings.json
{
  "env": {
    "CLAUDE_CODE_EFFORT_LEVEL": "max",
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `CLAUDE_CODE_EFFORT_LEVEL` | AI 的思考深度 | `medium` |
| `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` | 启用团队模式 | `0`（关闭） |
