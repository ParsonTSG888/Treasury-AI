# 进阶：5 个让 AI 更好用的底层概念——RAG、多模态、API、微调、本地部署

文 / Parson资金研习社

---

上一篇讲了 5 个基础关键词：Token、上下文窗口、幻觉、开源、温度。搞明白那 5 个，你用 AI 就不会"蒙着眼开"了。

但如果你已经用了一段时间 AI，大概率会碰到新的困惑：

"为什么 AI 总是给我通用答案，不了解我公司的制度？""为什么银行传来的扫描件 AI 读不了？""为什么 Claude Code 能调 DeepSeek 的模型，它们不是一家公司啊？""能不能把 AI 训练成'资金管理专家'？"

这些问题的背后，是 5 个进阶概念：RAG、多模态、API、微调、本地部署。

**上一篇是"仪表盘读法"，这一篇是"改装升级手册"——让你的 AI 从"能用"变成"好用"。**

---

## 一、RAG——让 AI 读了你的资料再回答

RAG，全称 Retrieval-Augmented Generation，中文叫"检索增强生成"。

名字很唬人，原理很简单：**先让 AI 去你指定的资料库里找相关内容，找到之后再回答你的问题。**

**打个比方：普通 AI 像一个刚入职的管培生，什么都懂一点，但不知道你公司的规矩。** 你问"我们公司的授信审批流程是什么"，它会给你一个"行业通用版"的答案——跟你公司的实际制度可能完全不一样。

RAG 的作用就是：在管培生回答之前，先让他翻一下你公司的《授信管理制度》《审批权限手册》，然后基于这些资料来回答。这样出来的答案就是"你们公司的版本"，而不是"教科书版本"。

**RAG 的工作流程分三步：**

第一步，**建索引**——把你的文档（授信制度、审批流程、银行合同模板、历史分析报告……）切成小段，让 AI 先"读"一遍，存成一种特殊的格式（叫"向量"，后面会简单解释）。

第二步，**检索**——你提问的时候，系统先去资料库里找最相关的几段内容。比如你问"集团授信审批的金额门槛是多少"，系统会自动找到授信管理制度里"审批权限"那一段。

第三步，**生成**——把找到的相关内容和你的问题一起发给 AI，AI 基于这些具体资料来回答。

**对资金人来说，RAG 解决了最大的痛点：AI 不了解你公司的业务。**

想想看，你公司的授信政策、利率审批权限、银行准入标准、内部资金定价规则——这些东西外面查不到，通用 AI 根本不可能知道。但通过 RAG，你可以把这些制度文档"喂"给 AI，让它变成一个"读过你公司所有制度的助手"。

**实操中怎么用 RAG？**

目前最简单的方案是 Claude Code + Obsidian 知识库。Obsidian 本身就是一个基于 Markdown 的知识库，Claude Code 可以直接读取里面的所有文件。你把公司的制度文件、历史报告、分析模板都放在知识库里，Claude Code 在回答问题时会自动搜索相关文件——这本质上就是 RAG 的工作方式。

更专业的方案是搭建专门的 RAG 系统（用向量数据库 + 大模型），但那需要技术人员介入，大多数资金部用 Claude Code + 文件夹的方式已经够了。

## 二、多模态——AI 不只会"读字"

多模态（Multimodal）是指 AI 能处理多种类型的信息：文字、图片、PDF 扫描件、音频、视频。

**早期的 AI 只能处理文字——你给它纯文本，它给你纯文本。** 但资金人的日常工作中，大量信息不是纯文本的：银行传来的盖章扫描件、手机拍的发票照片、会议录音、网页截图、Excel 图表截图……

多模态 AI 就是为了解决这个问题。

**资金场景中最实用的三个多模态能力：**

**第一，读 PDF 扫描件。** 银行发来的授信批复、贷款合同、利率调整通知，很多时候是盖章扫描的 PDF——本质上就是一张图片，不是可编辑的文字。普通 AI 读不了，但多模态 AI（比如 Claude）可以直接"看"扫描件，提取里面的文字内容。一份 30 页的扫描版授信批复，多模态 AI 半分钟就能把关键条款（额度、利率、期限、限制性条款）全部提取出来。

**第二，看图表。** 你截了一张 Excel 图表的截图，问 AI"这个趋势说明什么"——多模态 AI 能看懂图片里的折线、柱状图、数据标签，直接给出分析。不需要你把数据重新输入一遍。

**第三，听语音。** 银行客户经理打来电话说了一堆利率报价条件，你录了音。多模态 AI 可以把录音转成文字，再帮你整理成结构化的报价对比表。

**多模态对资金人最大的价值是：省掉了"把非文字信息转成文字"这一步。** 以前你得手动把扫描件的内容打出来、手动把图表数据抄下来、手动把录音内容整理出来——这些"转录"工作本身就是巨大的时间黑洞。多模态 AI 直接帮你跳过这一步。

**需要注意的限制：** 多模态 AI 对扫描件的识别准确率不是 100%。手写内容、模糊盖章、特殊格式的表格，识别率会明显下降。建议把多模态 AI 当作"初稿工具"——它帮你完成 80% 的转录工作，你负责校对剩下的 20%。

## 三、API——AI 的"万能插头"

API，全称 Application Programming Interface，中文叫"应用程序编程接口"。

**不需要记这个全称。你只需要知道一件事：API 就是让两个软件互相"对话"的通道。**

**打个比方：API 就像电源插座。** 你的电脑是"电器"，电网是"电源"，插座是连接两者的接口。有了标准插座，任何电器都能接上电网用电。API 也一样——有了标准接口，任何软件都能调用任何 AI 模型的能力。

**资金人为什么需要理解 API？因为 API 决定了你的 AI 工具能不能"联动"。**

一个典型的场景：你想让 Claude Code 用 DeepSeek 的模型来处理文件。Claude Code 是 Anthropic 的产品，DeepSeek 是另一家公司——它们怎么配合？答案就是通过 API。Claude Code 通过 API 调用 DeepSeek 的模型服务，就像你用一个万能插头把不同品牌的电器接上同一个电源。

再比如：你想让企业微信机器人自动调用 AI 来回答资金相关的问题。企业微信本身没有 AI 能力，但它可以通过 API 连接 DeepSeek 或 Claude 的模型——企业微信接收消息 → 通过 API 发给 AI 模型 → AI 处理后返回结果 → 企业微信把结果发给你。整个链条就是靠 API 串起来的。

**API 对资金人的三个实际影响：**

第一，**模型可以随便换。** 今天用 DeepSeek，明天觉得 Claude 更好，换个 API 地址就行——你的工作流程不用变。这就是为什么说"选工具比选模型重要"。

第二，**费用按调用量计算。** API 的计费方式是"用多少付多少"，按 Token 数量收费。这跟订阅制（比如 ChatGPT Pro 每月 200 美元）完全不同。对资金人来说，API 模式更灵活——用得少就花得少，不用每月固定支出。

第三，**安全可控。** 通过 API 调用，你可以精确控制哪些数据发给 AI、哪些不发。而且你可以选择国内的 API 服务商（比如 DeepSeek 官方 API），数据不出境，合规风险更低。

## 四、微调——把通用 AI 训练成"资金管理专家"

微调（Fine-tuning）是在已经训练好的大模型基础上，用你自己的数据再训练一遍，让模型更懂你的领域。

**打个比方：大模型像一个什么都会一点的通才。微调就是给这个通才做一次"岗前培训"——用你公司的历史报告、制度文件、业务数据来训练它，让它变成一个"懂你们公司业务的专家"。**

**微调跟 RAG 的区别是什么？**

RAG 是"临时翻资料"——AI 回答之前先去你的知识库里找相关内容，但模型本身没有变。微调是"把知识刻进脑子里"——模型本身的参数被修改了，它从底层就"学会了"你的业务逻辑。

| 维度 | RAG | 微调 |
|------|-----|------|
| 原理 | 回答前检索相关文档 | 用数据重新训练模型 |
| 成本 | 低（只需要整理文档） | 高（需要标注数据、算力、时间） |
| 更新频率 | 随时更新文档即可 | 每次更新都要重新训练 |
| 效果 | 回答基于文档内容，准确性取决于文档质量 | 回答风格和逻辑更贴近训练数据 |
| 适合场景 | 制度查询、文档分析、知识问答 | 特定格式输出、行业术语理解、专业判断 |

**资金人什么时候需要微调？**

说实话，大多数资金人不需要。RAG 已经能解决 90% 的"AI 不了解我公司"的问题。微调真正有价值的场景是：

- 你需要 AI 按照固定的格式输出（比如严格按照公司的资金日报模板，包含特定的表头、指标、注释格式）
- 你的行业有大量专业术语，通用模型经常理解错（比如"内部资金转移定价"被理解成普通的"转账"）
- 你需要 AI 做一些专业判断（比如根据历史数据判断某笔应收账款的坏账概率）

**微调的门槛很高：** 需要准备几百到几千条标注数据（输入-输出配对）、需要技术人员操作、需要 GPU 算力、每次制度更新都要重新训练。对绝大多数资金部来说，性价比不如 RAG。

**一个务实的建议：** 先用 RAG 把知识库搭好。如果用了半年发现"AI 的回答风格总是不符合要求"或者"专业术语理解偏差太大"，再考虑微调。不要一上来就搞微调——那是"进阶中的进阶"。

## 五、本地部署——把 AI 装在自己电脑上

本地部署是指把 AI 模型安装和运行在你自己的电脑或公司服务器上，而不是通过云端 API 调用。

**打个比方：云端 API 相当于"租用发电厂的电"，本地部署相当于"自己装了一台发电机"。** 发电厂的电方便、省事，但你得依赖发电厂，而且你用了多少电、什么时候用，发电厂都知道。自己的发电机完全独立，不依赖任何人，但你得自己买设备、自己维护。

**本地部署对资金人意味着什么？**

意味着你的数据从头到尾都不出你的电脑。授信合同、银行账户、子公司资金数据——所有信息只在本地处理，不经过任何外部服务器。这是数据安全的最高保障。

**本地部署需要什么条件？**

| 条件 | 要求 | 说明 |
|------|------|------|
| 硬件 | 至少 16GB 内存，推荐 32GB | 内存越大，能跑的模型越大 |
| 显卡 | 不强制，但有 NVIDIA 显卡速度更快 | 没有显卡也能跑，只是慢一些 |
| 存储 | 模型文件 5-20GB 不等 | DeepSeek V4 约 15GB |
| 软件 | Ollama 或 LM Studio | 一键安装，操作比装微信还简单 |
| 技术门槛 | 低 | 会下载软件、会点"安装"就行 |

**最简单的本地部署方案：Ollama + DeepSeek。**

Ollama 是一个免费的工具，专门用来在本地运行 AI 模型。安装过程大概是这样的：去 Ollama 官网下载安装包 → 双击安装 → 打开终端输入一行命令"ollama run deepseek-v4" → 等它下载模型（约 15GB）→ 下载完成后就能在终端里跟 AI 对话了。

整个过程大概 30 分钟到 1 小时（主要取决于网速），不需要写任何代码。

**本地部署的局限也得说清楚：**

第一，模型能力受硬件限制。你电脑的内存和显卡决定了能跑多大的模型。16GB 内存大概能跑 7B-14B 参数的模型，32GB 能跑更大的。但即使是最大的本地模型，跟 Claude Opus 这种云端顶级模型比，推理能力还是有差距。

第二，速度比云端慢。没有高端显卡的话，本地模型生成一段 500 字的回答可能需要 30 秒到 1 分钟，云端 API 通常 5 秒内就能返回。

第三，需要自己维护。模型更新了要重新下载，遇到问题得自己排查。不像云端 API，厂商帮你搞定一切。

**务实建议：** 如果公司信息部对数据安全有硬性要求（比如国企、金融机构），本地部署是唯一选择。如果数据安全压力不大，"云端 API + 数据脱敏"是更省事的方案。两者也可以混搭——敏感数据用本地模型处理，非敏感数据用云端 API，这是目前最主流的做法。

## 五个概念的关系

把这五个进阶概念串起来：

```
你想让 AI 更好用
    ↓ 先解决"AI 不了解我"
RAG（让 AI 读你的资料再回答）
    ↓ 再解决"AI 只能看文字"
多模态（让 AI 看图、听声音、读扫描件）
    ↓ 再解决"工具之间不联动"
API（让不同工具通过接口互相调用）
    ↓ 还想更进一步
微调（把行业知识训练进模型）
    ↓ 数据安全有硬性要求
本地部署（把 AI 装在自己电脑上）
```

**从实用角度看优先级：RAG > API > 多模态 > 本地部署 > 微调。**

大多数资金人先把 RAG 搞明白（用 Claude Code + 知识库文件夹就够了），再学会用 API 把不同工具串起来，就能覆盖 90% 的进阶需求。多模态、本地部署和微调属于"锦上添花"，等基础打好了再考虑。
