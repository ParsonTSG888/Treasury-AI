【AI系列·第20篇】CLAUDE.md——写好这一份文件，AI的产出翻倍

文 / Parson资金研习社

---

用了几个月 Claude Code，你可能发现一个规律：

同样一个任务，有的人跑一遍就出结果，有的人要反复调教五六遍。差距不在模型，在配置。

**Claude Code 的"配置"不是 API Key 和模型选择——那些只是基础。真正的配置，是 `CLAUDE.md`。**

这个文件决定了 AI 的"默认行为"——它用什么语气说话、遵循什么规则、优先做什么检查、避免什么错误。写好了，AI 的产出质量直接翻倍；没写或者写得不好，每次对话都在浪费 token 重复交代背景。

---

## 一、CLAUDE.md 是什么

一句话：**CLAUDE.md 是 Claude Code 的"项目说明书"，每次对话开始时自动加载。**

你可以把它理解为 Claude Code 的 `.bashrc` 或 `.gitconfig`——启动时自动读取，影响后续所有行为。

它不需要你手动调用。你打开 Claude Code 的那一刻，它就已经在读 `CLAUDE.md` 了。

存放位置有三种：

| 位置 | 文件路径 | 作用范围 |
|------|---------|---------|
| 用户级 | `~/.claude/CLAUDE.md` | 所有项目通用 |
| 项目级 | 项目根目录下的 `CLAUDE.md` | 仅当前项目 |
| 子目录级 | 子目录下的 `CLAUDE.md` | 仅当操作涉及该子目录 |

**优先级：子目录 > 项目 > 用户级。** 三者会合并，但同名规则以更具体的为准。

## 二、为什么 CLAUDE.md 如此重要

### 原因一：省掉重复交代

没有 `CLAUDE.md` 的人，每次对话都要说：

> "帮我生成一份资金日报，格式是这样的：第一列是子公司名称，第二列是银行账号，第三列是今日余额……币种统一用 CNY……金额单位是万元……小数点后两位……"

有 `CLAUDE.md` 的人，只需要说：

> "帮我生成今天的资金日报。"

因为格式、单位、币种这些规则，已经写在 `CLAUDE.md` 里了。

**每次省 2-3 分钟的提示词输入，一天 10 次对话就是 20-30 分钟。**

### 原因二：减少 AI 犯错

没有约束的 AI 是"自由发挥"的——它可能把金额写成元而不是万元，把币种写成 USD 而不是 CNY，把表格格式换成它"觉得好看"的样子。

`CLAUDE.md` 相当于给 AI 画了一条"红线"——在这些范围内你可以自由发挥，但这些规则不能违反。

### 原因三：团队统一标准

团队里 5 个人都用 Claude Code，每个人的提示词风格不一样，出来的报告格式五花八门。

把 `CLAUDE.md` 提交到 Git 仓库，所有人共享同一套规则。不管谁跑，出来的报告格式一样、口径一致、质量稳定。

## 三、CLAUDE.md 的五个核心模块

一个好的 `CLAUDE.md`，必须包含五个模块。少了任何一个，效果都会打折扣。

### 模块一：项目背景

告诉 AI "你在做什么"。

```markdown
## 项目背景

这是一个集团资金管理知识库项目，包含 1990+ 篇专业文章。
知识库用 Obsidian + WPS 知识库管理，内容覆盖资金日报、银行授信、
外汇管理、供应链金融等领域。
```

**为什么重要：** AI 不知道你的项目背景，它只能根据当前对话推断。明确告诉它，避免它做出不相关的假设。

### 模块二：技术栈与工具

告诉 AI "你用什么工具"。

```markdown
## 技术栈

- 编辑器：Obsidian
- AI 工具：Claude Code + DeepSeek V4
- 知识库：WPS KnowledgeBase
- 图表：ECharts 5.x
- 数据格式：Markdown + Excel
```

**为什么重要：** 如果你不告诉 AI 用 ECharts，它可能默认用 D3.js 或者 Matplotlib。技术栈写清楚，AI 的选择更精准。

### 模块三：输出规范

告诉 AI "输出什么格式"。

```markdown
## 输出规范

### 资金日报
- 格式：Markdown 表格
- 金额单位：万元（保留两位小数）
- 币种：CNY（不要写"人民币"）
- 表头：子公司名称 | 银行账号 | 今日余额 | 昨日余额 | 变动金额 | 备注

### 文章标题
- 格式：【AI系列·第XX篇】标题
- 标题长度：15-25 字

### 代码输出
- 语言：Python 3.11+
- 注释：中文
- 错误处理：必须包含 try-except
```

**为什么重要：** 这是最直接影响产出质量的部分。没有输出规范，AI 会"自由发挥"——每次的格式、单位、风格都不一样。

### 模块四：安全规则

告诉 AI "什么不能做"。

```markdown
## 安全规则

- 绝对不要在输出中包含真实的银行账号、API Key、密码
- 不要把 `.env` 文件的内容输出到对话中
- 不要修改 `production/` 目录下的任何文件
- 涉及数据库操作，只允许 SELECT，不允许 INSERT/UPDATE/DELETE
```

**为什么重要：** 这是"红线"。AI 可能不知道什么是敏感数据，明确告诉它哪些绝对不能碰。

### 模块五：常见错误与纠正

告诉 AI "哪些错误你之前犯过"。

```markdown
## 常见错误

- ❌ 把"万元"当成"元"来计算 → ✅ 始终检查原始数据的单位
- ❌ 把所有子公司混在一起汇总 → ✅ 先按子公司分组，再逐个校验
- ❌ 生成的表格用英文表头 → ✅ 必须用中文表头
```

**为什么重要：** 这是"经验传承"。你踩过的坑，写进 `CLAUDE.md`，AI 就不会再踩。

## 四、财资人的 CLAUDE.md 模板

下面是一个完整的模板，你可以直接复制修改：

```markdown
# CLAUDE.md — 集团资金管理知识库

## 项目背景

集团资金管理知识库，包含 1990+ 篇专业文章，覆盖资金日报、银行授信、
外汇管理、供应链金融、资金监控等领域。面向财资从业者，内容以实战为主。

## 技术栈

- AI 工具：Claude Code + DeepSeek V4 / MiMo v2.5
- 知识库：WPS KnowledgeBase（通过 MCP 连接）
- 本地笔记：Obsidian
- 图表：ECharts 5.x
- 数据格式：Markdown + Excel
- 安全：敏感数据不上传公共 AI 平台

## 输出规范

### 金额
- 单位：万元（保留两位小数）
- 币种：CNY（不要写"人民币"或"RMB"）
- 负数表示：支出/流出用负数，收入/流入用正数

### 表格
- 表头：中文
- 对齐：数字右对齐，文字左对齐
- 序号列：不加序号列（除非明确要求）

### 文章
- 标题格式：【AI系列·第XX篇】标题
- 标题长度：15-25 字
- 风格：实战导向，少用"建议关注"这种废话
- 结构：先说问题，再说方案，最后给操作步骤

### 代码
- 语言：Python 3.11+
- 注释：中文
- 错误处理：必须包含 try-except
- 文件编码：UTF-8

## 安全规则

- 绝对不要输出真实的银行账号、API Key、密码
- 不要把 `.env` 文件内容输出到对话
- 不要修改 `production/` 目录下的文件
- 数据库操作只允许 SELECT

## 常见错误

- ❌ 把"万元"当成"元"计算 → ✅ 始终检查原始数据单位
- ❌ 把所有子公司混在一起汇总 → ✅ 先按子公司分组
- ❌ 生成表格用英文表头 → ✅ 必须用中文表头
- ❌ 用"建议关注""总体来看"这种套话 → ✅ 给具体数字和结论
```

## 五、进阶技巧

### 技巧一：用"规则标签"控制行为

在 `CLAUDE.md` 里用标签标记规则的适用范围：

```markdown
## 规则

- [always] 金额必须保留两位小数
- [article] 文章标题用中文
- [code] 代码必须包含错误处理
- [data] 数据库操作只允许 SELECT
```

`[always]` 表示所有场景都适用，`[article]` 表示只在写文章时适用，`[code]` 表示只在写代码时适用。

这不是 Claude Code 的官方功能，而是一种约定——你可以在提示词里引用这些标签，AI 会理解。

### 技巧二：用"示例"代替"描述"

与其说"表格要简洁、专业、对齐"，不如直接给一个示例：

```markdown
## 输出示例

资金日报表格：

| 子公司 | 银行 | 余额（万元） | 变动 |
|--------|------|-------------|------|
| 子公司A | 工商银行 | 1,258.47 | +123.50 |
| 子公司B | 建设银行 | 3,421.00 | -500.00 |
```

**示例比描述有效 10 倍。** AI 看到示例，输出格式会高度一致。

### 技巧三：分文件管理

如果 `CLAUDE.md` 内容太多（超过 200 行），可以拆分成多个文件：

```
项目根目录/
├── CLAUDE.md              # 主文件，包含核心规则
├── .claude/
│   ├── rules/
│   │   ├── data-format.md  # 数据格式规范
│   │   ├── article-style.md # 文章风格规范
│   │   └── security.md     # 安全规则
│   └── examples/
│       └── daily-report.md # 资金日报示例
```

在主文件里引用：

```markdown
## 相关规则

- 数据格式：见 `.claude/rules/data-format.md`
- 文章风格：见 `.claude/rules/article-style.md`
- 安全规则：见 `.claude/rules/security.md`
```

### 技巧四：定期迭代

`CLAUDE.md` 不是写一次就完事的。

每次你发现 AI 犯了一个新错误，就加一条规则。每次你的工作流程变了，就更新技术栈。每次你发现某个规则多余了，就删掉。

**建议每月 review 一次 `CLAUDE.md`，保持它跟你的实际工作流一致。**

## 六、常见错误

### 错误一：写得太长

有人把 `CLAUDE.md` 写成了几百行的"百科全书"——什么规则都往里塞。

问题：`CLAUDE.md` 每次对话都会加载，太长会占用大量上下文窗口（token），导致留给实际任务的空间变少。

**建议：核心规则控制在 100 行以内。** 详细的参考资料放到单独的文件里，在需要时再加载。

### 错误二：写得太抽象

"输出要专业""表格要美观""代码要规范"——这些规则 AI 读了等于没读。

**建议：用具体的、可检查的规则。** "金额保留两位小数"比"金额要精确"好，"表头用中文"比"表头要规范"好。

### 错误三：不写安全规则

没有安全规则，AI 可能在输出中暴露你的 API Key、数据库密码、或者真实的银行账号。

**建议：安全规则是必写项，不是可选项。**

### 错误四：写了不更新

半年前写的 `CLAUDE.md`，技术栈还是"ChatGPT + Notion"，现在已经换了 DeepSeek + Obsidian。旧规则反而干扰 AI 的判断。

**建议：每次工具或流程变更，同步更新 `CLAUDE.md`。**

## 七、和 Skill 的分工

第 7 篇讲了 Skill，这一篇讲了 `CLAUDE.md`。两者有什么区别？

| 维度 | CLAUDE.md | Skill |
|------|-----------|-------|
| 作用范围 | 全局（所有对话生效） | 特定任务（触发时生效） |
| 内容 | 通用规则、背景、约束 | 具体的工作流程 |
| 类比 | 公司的《员工手册》 | 某个岗位的《操作 SOP》 |
| 触发方式 | 自动加载 | 关键词触发或手动调用 |
| 适合放什么 | 输出格式、安全规则、技术栈 | "怎么做资金日报"的具体步骤 |

**两者不冲突，互相配合。** `CLAUDE.md` 定义"通用规则"，Skill 定义"具体流程"。AI 先读 `CLAUDE.md` 了解全局，再读 Skill 了解当前任务。
