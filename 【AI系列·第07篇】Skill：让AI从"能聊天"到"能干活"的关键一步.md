# Skill：让 AI 从"能聊天"到"能干活"的关键一步

文 / Parson资金研习社

---

前面几篇，把主流大模型都盘了一遍——DeepSeek、Claude、MiMo，哪个便宜、哪个能打、怎么接入 Claude Code。

但有一个问题一直没解决：**模型再强，你每次都要从零开始教它干活。**

"帮我生成一份资金日报"——结果格式不对，重来。
"帮我做一份应收账款账龄分析"——它漏了逾期分级，重来。
"帮我把这 50 个子公司的数据汇总"——它把币种搞混了，重来。

问题不在模型，在你。**你没有给它一套标准作业流程（SOP）。**

这就是 Skill 的价值——把你的专业经验"封装"成 AI 能反复执行的标准动作。

---

## 一、Skill 是什么

Skill 不是"提示词"。

提示词是你临时写的一段话，用完就扔。Skill 是你精心打磨的一套**可复用的专业流程**，保存成文件，随时调用。

打个比方：

- 提示词 = 你口头跟新员工说"帮我把这个表格整理一下"
- Skill = 你给新员工发了一份《资金日报标准化操作手册》，里面写清楚了每一步怎么做、哪些格式必须遵守、哪些错误必须避免

**提示词是一次性的指令，Skill 是可传承的 SOP。**

在 Claude Code 里，Skill 就是一个 Markdown 文件，放在项目的 `Skills/` 目录下。当你输入 `/skill-name` 或者说了触发词，Claude Code 会自动加载这个文件，按照里面的流程执行。

---

## 二、为什么 Skill 比"会写提示词"重要 10 倍

很多人觉得"会用 AI"就是"会写提示词"。这是 2024 年的认知。

2026 年的认知是：**会写 Skill 的人，和只会写提示词的人，效率差距是 10 倍。**

原因有三个：

**1. 一致性**

提示词每次写得不一样，AI 的输出也不一样。今天生成的资金日报格式是 A，明天变成 B，后天变成 C。Skill 把格式锁死了——每次调用，输出一模一样。

**2. 可传承**

提示词在你脑子里，你请假了没人能用。Skill 是文件，任何人都能调用。你花 3 天打磨的资金日报流程，团队里每个人都能一键执行。

**3. 可迭代**

提示词用完就丢了，下次还得从头写。Skill 可以持续优化——发现 AI 漏了某个检查项，改一下 Skill 文件就行，下次自动执行新版本。

---

## 三、Skill 的标准架构：五个模块

一个好的 Skill，必须包含五个模块。少一个，效果打折扣。

### 模块一：元数据（Frontmatter）

文件头部的 YAML 块，告诉系统这个 Skill 是什么、什么时候触发。

```yaml
---
name: daily-treasury-report
description: 集团资金日报自动生成器
series: Skills
created: 2026-05-28
tags:
  - 资金日报
  - 数据处理
---
```

`name` 是 Skill 的唯一标识符，`description` 是一句话说明用途。这两个字段决定了 Claude Code 能不能"找到"你的 Skill。

### 模块二：触发词（Trigger Words）

告诉 AI "什么时候该用这个 Skill"。不是你手动调用，而是你说出某些关键词时，AI 自动识别并加载。

```
## 触发词

- 生成资金日报
- 做日报
- 资金日报汇总
- 日报模板
- 今天的资金情况
```

触发词的设计原则：**覆盖用户最可能说的 5-10 种表达方式**。别只写一个"生成日报"，用户可能说"做份日报""今天的资金情况怎么样""把今天的头寸整理一下"——都要覆盖到。

### 模块三：工作流程（Workflow）

这是 Skill 的核心——**一步一步告诉 AI 该做什么**。

以"资金日报生成"为例：

```
## 工作流程

### Step 1：收集原始数据
检查用户是否提供了 Excel 文件。
如果提供了，读取文件并识别数据结构。
如果没有提供，提示用户提供当日资金数据。

### Step 2：数据清洗与标准化
- 统一币种编码（CNY/USD/EUR/HKD）
- 统一金额单位（万元）
- 识别并标记异常值（单笔超过 1000 万、日变动超过 30%）
- 合并同一子公司的多币种账户

### Step 3：生成日报
按照标准模板输出，必须包含以下模块：
1. 集团资金头寸总览（按币种）
2. 子公司资金明细（按区域分组）
3. 当日大额收支明细（单笔 500 万以上）
4. 异常事项标注
5. 授信额度使用情况
```

工作流程的设计原则：

- **每一步只做一件事**——别把"收集数据"和"生成日报"塞在同一步
- **明确输入和输出**——这一步需要什么输入？产出什么结果给下一步？
- **写清判断逻辑**——如果数据不全怎么办？如果触发了异常怎么办？

### 模块四：约束条件（Constraints）

告诉 AI "哪些事不能做"。这是防止 AI 犯错的关键。

```
## 约束条件

- 金额单位统一为"万元"，保留两位小数
- 币种必须使用 ISO 4217 编码（CNY/USD/EUR），禁止使用"人民币""美元"等中文
- 异常值必须标注，不能自动删除或修正
- 数据来源必须标注（SAP/网银/手工填报）
- 禁止编造缺失数据，缺失项标注为"N/A"
- 子公司排序按区域分组：亚太→欧洲→美洲→其他
```

约束条件的原则：**越具体越好**。"数据要准确"是废话；"金额单位统一为万元，保留两位小数"才是有效约束。

### 模块五：输出模板（Template）

告诉 AI "最终输出长什么样"。

```
## 输出模板

### 标题
{日期} 集团资金日报

### 第一部分：头寸总览
| 币种 | 期初余额 | 当日收入 | 当日支出 | 期末余额 | 日变动率 |
|------|---------|---------|---------|---------|---------|

### 第二部分：子公司明细
按区域分组，每个区域一个表格。

### 第三部分：大额收支
单笔 500 万以上的收支明细。

### 第四部分：异常事项
- 标红：日变动超过 30% 的账户
- 标黄：余额低于安全垫的账户

### 第五部分：授信使用
| 银行 | 总额度 | 已用额度 | 可用额度 | 使用率 |
```

输出模板的原则：**给 AI 一个"填空"的框架，而不是让它自由发挥**。

---

## 四、"不超过 500 字"原则

很多人写 Skill 犯的最大错误：**写太多。**

Skill 不是论文，不是操作手册的全文。它是 AI 的"速查表"——AI 读完这 500 字，就知道该怎么干活。

为什么是 500 字？

- AI 的上下文窗口是有限的，Skill 太长会挤占留给实际任务的空间
- Skill 越长，AI 越容易"迷路"——找不到重点
- 500 字足够写清楚一个标准流程，多了就是废话

**怎么做到 500 字以内？**

1. 砍掉所有"为什么"——Skill 告诉 AI "怎么做"，不需要解释"为什么这么做"
2. 砍掉所有示例——除非这个示例是 AI 必须模仿的格式
3. 砍掉所有背景知识——AI 已经知道了，不需要你再教
4. 只保留"必须遵守的步骤"和"绝对不能犯的错"

## 五、从零写一个 Skill：以"应收账款账龄分析"为例

来看一个真实的 Skill 是怎么从零构建的。

**第一步：明确任务**

目标：根据应收账款明细数据，生成按客户/币种/账龄分级的分析报告。

**第二步：列出 AI 必须做的事**

1. 读取应收账款明细 Excel
2. 按账龄分级：0-30 天、31-60 天、61-90 天、91-180 天、180 天以上
3. 按客户汇总：每个客户的应收账款总额和加权平均账龄
4. 按币种汇总：各币种的应收账款占比
5. 标记逾期客户：超过合同账期的客户标红
6. 计算坏账准备：按账龄分级计提比例估算坏账准备

**第三步：列出 AI 绝对不能犯的错**

- 不能把"到期日"和"账龄"搞混——账龄 = 报告日 - 开票日，不是报告日 - 到期日
- 不能遗漏任何一笔应收账款
- 不能自己编造计提比例——必须使用用户提供的比例或默认行业标准
- 不能改变原始数据——只做分析，不改源数据

**第四步：写出工作流程**

```
## 工作流程

### Step 1：读取数据
检查用户是否提供了应收账款明细文件。
必需字段：客户名称、开票日期、到期日、金额、币种、合同账期。
缺失字段提示用户补充。

### Step 2：数据处理
- 计算每笔应收账款的账龄（报告日 - 开票日期）
- 按账龄分级：0-30/31-60/61-90/91-180/180+ 天
- 标记逾期状态：当前日期 > 到期日 为逾期
- 按币种折算为人民币（使用报告日汇率）

### Step 3：生成分析报告
输出三个维度的分析：
1. 按账龄分级汇总（含占比）
2. 按客户汇总（按逾期金额降序，Top 20）
3. 按币种汇总（含汇率折算说明）
```

**第五步：加上约束条件和输出格式**

```
## 约束条件
- 账龄 = 报告日 - 开票日（不是到期日）
- 金额保留两位小数，单位为"万元"
- 逾期客户标红，但不删除
- 汇率使用报告日央行中间价
- 缺失数据标注"N/A"，不编造

## 输出格式
- 按账龄分级：表格 + 占比柱状图描述
- 按客户：Top 20 表格 + 逾期金额
- 按币种：饼图描述 + 汇率折算说明
```

整合起来，就是一个完整的 Skill 文件。总字数控制在 500 字以内，AI 读完就知道该怎么做。

## 六、Skill 的使用方式

在 Claude Code 里，Skill 有三种调用方式：

**方式一：斜杠命令**

```
/应收账款账龄分析
```

直接调用指定的 Skill。

**方式二：触发词自动识别**

```
"帮我做一份应收账款账龄分析"
```

AI 识别到"应收账款"和"账龄分析"两个触发词，自动加载对应的 Skill。

**方式三：在对话中引用**

```
按照 Skills/应收账款账龄分析.md 的流程执行
```

显式指定 Skill 文件路径。

## 七、Skill 的迭代优化

Skill 不是写完就不管了。好的 Skill 需要持续迭代。

**迭代的触发条件：**

- AI 输出不符合预期——检查是 Skill 的流程描述不够清晰，还是约束条件有漏洞
- 任务需求变了——比如原来只需要按账龄分级，现在还要加上坏账准备计提
- 发现了新的"坑"——比如 AI 总是把"开票日"和"入账日"搞混，就在约束条件里加一条

**迭代的方法：**

1. 记录问题：AI 哪里做错了？
2. 定位原因：是流程没写清楚？还是约束没覆盖到？
3. 修改 Skill：加上缺失的步骤或约束
4. 测试验证：用同样的输入再跑一遍，看是否修复

**这就是"去AI化"的真正含义**——不是让 AI 不像 AI，而是让 AI 的输出稳定地像"有经验的从业者写的"。Skill 就是那个"有经验的从业者"的数字化身。

---

## 八、写在最后

前面几篇讲了模型——DeepSeek、Claude、MiMo，哪个便宜、哪个能打。

但模型只是引擎。**Skill 才是方向盘。**

没有 Skill，你每次都在重新发明轮子。有了 Skill，你是在铺设铁轨——铺好了，列车自动跑。

一个财资人的 Skill 库，就是他的**专业能力的数字化沉淀**。你花 10 年积累的资金管理经验、日报流程、分析框架，都可以封装成 Skill，让 AI 替你执行。

**你负责定义"什么是对的"，AI 负责"每次都做对"。**

这才是 AI 时代财资人的正确姿势。
